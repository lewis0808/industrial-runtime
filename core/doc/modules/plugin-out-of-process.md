# 进程外插件隔离 — 方案探索（Exploration）

> 状态：**探索 / 未实现**。本文梳理动机、架构选型与取舍，作为后续决策与排期的输入，
> 不含落地代码。对应 roadmap P2 #18 的「关键场景探索进程外插件隔离」。

## 1. 为什么需要

当前插件**进程内**加载（`LoadLibrary`/`dlopen`），与运行时共享地址空间。已有的 C-ABI
异常隔离（`guardedCall` + `noexcept` thunk，见 [plugin-system.md](plugin-system.md)）只能挡住
**跨 DLL 的 C++ 异常**；挡不住**硬故障**：

- 段错误 / 野指针写坏运行时内存；
- `abort()` / `std::terminate` / 栈溢出直接带走整个进程；
- 第三方设备 SDK（snap7、相机/雷达厂商库）内部崩溃、死锁、句柄泄漏；
- 死循环 / 长阻塞拖垮线程（在途排空也会一直等）。

对**关键场景**（产线不可停、单插件不可信/不稳定、需独立升级重启），需要更强的隔离边界：
**进程**。崩溃只波及子进程，运行时存活；卸载 = 杀进程，天然干净。

## 2. 目标 / 非目标

**目标**
- 进程级故障隔离：子进程崩溃不影响 core 与其它插件。
- 与现有插件 ABI **语义一致**：插件作者代码（`IPlugin` / `createPlugin`）尽量不改，
  数据面仍是 `pushTag/pushEvent/pushStream` + 写回。
- 复用现有热卸载语义：`unload(id)` 对进程外插件 = 优雅停 + 杀进程（更简单、更彻底）。
- **opt-in、可混部**：默认仍进程内（零开销）；按插件声明「需隔离」才进程外。

**非目标**
- 不追求跨机分布式（那是另一个量级，IRSP 数据面才管对外）。
- 不在本轮替换进程内模型——进程外是**附加的隔离层**，不是默认。

## 3. 总体架构

```
            ┌─────────────────────── Runtime 进程 ───────────────────────┐
            │  RuntimeEngine / TagEngine / EventBus / ...                 │
            │        ▲ pushTag/Event/Stream      │ writeTag               │
            │  ┌─────┴───────────────────────────▼──────┐                 │
            │  │  PluginHost (现有 C-ABI 宿主)            │                 │
            │  └─────▲───────────────────────────────────┘                 │
            │        │ 进程内插件（默认，函数调用）                          │
            │  ┌─────┴──────────┐   ┌──────────────────────────────┐       │
            │  │ in-proc plugin │   │  OutOfProcHost / Supervisor   │       │
            │  └────────────────┘   │  - 启动/监护子进程            │        │
            │                       │  - IPC 编解码 + 路由           │       │
            │                       │  - 崩溃检测 + 重启策略         │       │
            │                       └───────────────┬──────────────┘       │
            └───────────────────────────────────────│──────────────────────┘
                                                     │ IPC（见 §5）
                       ┌─────────────────────────────▼─────────────────────┐
                       │            Plugin Host 子进程（每插件/每组）         │
                       │  ir-plugin-host.exe <dll> <config>                 │
                       │  - 进程内加载真实插件 DLL（复用现有 PluginManager） │
                       │  - 把宿主 API 转成 IPC 调用回 Runtime               │
                       └────────────────────────────────────────────────────┘
```

要点：**子进程内仍用现有的进程内加载机制**（`PluginManager` + `PluginHost`）。区别在于子进程里
的 `PluginHost` 不直接写 `TagEngine`，而是把 `pushTag/...` 经 IPC 发回 Runtime；Runtime 侧的
`OutOfProcHost` 收到后调真正的 `RuntimeApi`。**插件作者代码完全不变**——隔离对插件透明。

## 4. 与现有 ABI 的关系

C-ABI 的五个生命周期 + 四个数据面调用，映射为 IPC 报文：

| 现有调用 | 方向 | 进程外形态 |
|----------|------|-----------|
| `createPlugin/init/start/stop/destroy` | Runtime → 子进程 | 控制报文（请求/应答）；`destroy` 后等子进程退出 |
| `push_tag/push_event` | 子进程 → Runtime | 单向数据报文（可批量/异步） |
| `push_stream` | 子进程 → Runtime | 走**高带宽通道**（共享内存，见 §5） |
| `register_writer` + 写回 `write()` | 双向 | 注册时报前缀；写回为请求/应答（同步「已受理」语义不变） |

owner / 引用计数排空在**进程内**仍需要（子进程内的 `write()` 仍是函数指针）；Runtime 侧对
进程外插件的「卸载」简化为：停止派发 → 发 `stop` → 超时则 `TerminateProcess`/`SIGKILL`，
无需排空 Runtime 自身的函数指针（边界是 IPC，不是裸指针）。

## 5. IPC 传输选型

Tag/Event/控制是小报文、低频~中频；Stream 是大块、高带宽、需背压——**两类分开承载**：

| 通道 | 候选 | 取舍 |
|------|------|------|
| 控制 + Tag + Event | **本机 socket**（Win: named pipe / AF_UNIX；Linux: AF_UNIX）| 跨平台成熟、有连接语义、断开即知子进程死；吞吐足够小报文 |
| Stream（图像/点云） | **共享内存环形缓冲 + 信号量** | 零拷贝、高吞吐；socket 传大帧会拷贝+系统调用开销过高 |

编码：控制/Tag/Event 可**复用 irsp1 编解码**（已有、自描述、跨语言），或选 MsgPack（与
IRSP V2 演进一致）。Stream 走共享内存时只在控制通道传「帧就绪 + 偏移/长度」元数据，数据本体
在共享内存里，避免序列化大块。

> 注意：`IrPluginString`/payload 现在是「非拥有、调用内同步拷贝」的语义；跨进程后所有数据
> 必须**值传递**（拷进 IPC 缓冲或共享内存），不能再传指针。这对子进程侧的封送层是主要改动点。

## 6. 故障处理与监护

- **崩溃检测**：控制通道断开（pipe/socket EOF）或子进程退出码非 0 → 判定崩溃。
- **重启策略**：每插件可配 `none` / `on-crash` / `always` + 退避（指数 backoff + 上限），
  避免崩溃循环打满 CPU。重启即重新 `createPlugin/init/start`。
- **健康检查**：控制通道周期 ping / 看门狗；子进程无响应超时 → 视为死，杀+按策略重启。
- **资源回收**：子进程持有的句柄/内存随进程退出由 OS 回收——这正是进程隔离相对 DLL 卸载的
  根本优势（DLL 卸载难保第三方 SDK 不泄漏）。
- **数据面影响**：子进程崩溃期间，其负责的 Tag 停更、写回返回 `NOT_FOUND`/`ERR`；可选地
  发一条 `critical` 事件通知订阅者（运维可见）。

## 7. 背压（Stream 专属）

共享内存环形缓冲满时的策略需明确（与 EventBus 的溢出策略呼应）：
- **丢最旧**（实时视频/点云常用，保新鲜度）；
- **阻塞生产者**（子进程，保不丢但可能拖慢采集）；
- **丢并计数 + 指标**（可观测）。
按 Stream 类型可配，默认丢最旧 + 计数。

## 8. 分阶段落地建议

1. **M0（本仓库已完成）**：进程内热卸载/reload + owner 排空 + 独立 admin 控制面通道
   （`admin/`，命名管道/AF_UNIX 上的 `PLUGIN` 命令）——为进程外复用同一套生命周期/控制面语义打底。
2. **M1 子进程宿主可执行**：`ir-plugin-host` 可执行，进程内加载一个 DLL，控制通道跑通
   `create/init/start/stop`，Tag/Event 经 socket 回流。先不做 Stream/重启。
3. **M2 监护 + 重启 + 写回**：`OutOfProcHost` 接 `PluginManager` 同一管理面（`unload`=杀进程），
   写回请求/应答，崩溃检测 + 重启策略。
4. **M3 Stream 共享内存通道 + 背压**：高带宽零拷贝，按类型背压策略。
5. **M4 opt-in 声明 + 混部**：插件经 `getPluginInfo` 扩展位 / config 声明 `isolation: process`，
   `PluginManager` 据此选进程内 or 进程外，二者对 IRSP/Runtime 透明。

## 9. 取舍与代价（必须正视）

- **性能**：进程内是函数调用（纳秒级、零拷贝）；进程外每次数据面调用是 IPC（微秒级 + 拷贝）。
  故**默认进程内**，仅对「需隔离 / 不可信 / 易崩」的插件 opt-in 进程外。Tag 频率高的插件
  尤其要权衡（可批量/合并上行缓解）。
- **复杂度**：监护、重启、IPC 编解码、共享内存生命周期、跨进程数据所有权——是一个独立子系统，
  工作量远大于进程内热卸载。需有真实「关键场景」需求驱动再投入，避免过度设计。
- **跨平台**：named pipe vs AF_UNIX、共享内存 API、进程创建/终止/退出码各平台不同，需抽象层。
- **调试**：跨进程调试比单进程难（需附加到子进程、跨进程日志关联 id）。

## 10. 开放决策点

1. 隔离粒度：**每插件一进程**（隔离彻底、进程多）vs **每组/每类一进程**（省资源、组内不隔离）？
2. 编码：复用 irsp1 还是 MsgPack（与 IRSP V2 对齐）？是否控制面与数据面用不同编码？
3. Stream 共享内存的生命周期与安全（子进程崩溃时共享内存如何安全回收/防读到半写帧）。
4. opt-in 声明位放哪：`getPluginInfo` 结构追加字段（ABI 末尾追加，兼容）还是纯 config？
5. 与 README §7「独立 admin 控制面」的关系：进程外监护的控制是否也走该 admin 通道？
